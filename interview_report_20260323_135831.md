## 二、参考答案（本场涉及要点）

### MCP 与 REST

- MCP 是 **Agent↔工具/资源** 的标准协议（tools/resources/prompts），不仅是「JSON Schema 统一」；Stdio 场景下零端口、日志与 MCP 消息分离等也是常见考点。
- 相对 REST：减少 **每个 Host/模型各适配一套** 的重复，生态里 Client 可复用；REST 仍可自建，但集成成本高。

### P50 / P99

- **P50**：延迟中位数；**P99**：高分位长尾。个人单机可先看 P50 稳定性，再看 P99 是否偶发卡顿。

### 精排候选规模

- 两路召回后会 **融合（如 RRF）**，候选数 **通常不是两路简单相加**，还有去重与 `top_k_final` 等裁剪。
- K 过大：**Cross-Encoder/LLM 精排成本**、**上下文长度与噪声**、尾部文档相关性下降，均为合理扣分点。

## 三、简历包装点评

### 包装合理 ✅

- **「MCP + 结构化引用」**：与项目 `query_knowledge_hub` 返回 `content` / `structuredContent` 方向一致。
- **「混合检索 + 精排」**：与架构主干一致（具体稀疏实现见下）。

### 需对齐实现 ❌ / ⚠️

- **「Dense + BM25」**：默认 **SQLite 后端** 下稀疏一路多为 **FTS5**，口语可称「稀疏关键词一路」，避免与面试官对「独立 BM25 索引文件」产生偏差。
- **「15%–25% 提升」**：本场未追问测量方法；真实面试需准备 **baseline、数据集规模、指标名（如 Recall@K）**。

### 改进建议

1. MCP：背一句 **「协议边界 + Stdio 适用场景」**，少只强调 schema。
2. 延迟：能说出 **P50 vs P99 选型**（长尾敏感业务更看 P99）。
3. 检索数字：**融合去重**后再谈进入精排的规模；对照 `config/settings.yaml` 的 `top_k_dense/sparse/final`、`rerank.top_m`。

**优势**：精排与上下文噪声、成本意识清楚；愿意暴露使用场景（个人/单用户）。
